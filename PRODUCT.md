# Product

## Register

brand

## Users

Frontend engineers, tech leads, and design-system maintainers who need to compare static CSS-in-JS options without reading multiple documentation sets end to end. They are likely evaluating a new styling architecture, a migration away from runtime CSS-in-JS, or a component-library override model.

## Product Purpose

Static Style Lab explains practical tradeoffs between vanilla-extract, StyleX, MUI Pigment CSS, and Emotion. Success means a reader can understand where each tool wins, where it becomes risky, and how composition and overrides behave before they commit a team to a styling system.

## Brand Personality

Technical, research-driven, and opinionated. The interface should feel like a working field guide for frontend architecture, not a generic SaaS landing page.

## Anti-references

Avoid generic startup landing pages, oversized marketing cards, decorative gradients, warm beige monoculture, and vague claims that hide implementation tradeoffs. Avoid treating "static site" and "client-rendered static assets" as the same thing.

## Design Principles

- Practice the subject matter: prefer build-time output, inspectable artifacts, and minimal runtime work.
- Show the mechanism: make composition, override order, and dynamic-value pressure visible.
- Keep the comparison honest: name adoption risk, project status, and migration cost directly.
- Design for scanning first, then deep reading: let a technical reader jump from summary to evidence to examples without losing context.
- Preserve live learning where it matters: interactive demos are valuable when they explain a model that prose alone would flatten.

## Accessibility & Inclusion

Target WCAG AA contrast for text and controls. Preserve keyboard access for all controls and links. Respect reduced-motion preferences. Avoid relying on color alone to explain demo states or tool fit.
